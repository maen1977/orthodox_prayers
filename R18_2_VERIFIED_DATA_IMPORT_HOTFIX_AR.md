# إصلاح R18.2 لاستيراد فرع verified-data القديم

## سبب تكرار الخطأ

إصلاح R18.1 نظّف مجلد المصدر أثناء إنشاء بيانات اليوم، لكن Workflow **Build** كان يجلب فرع `verified-data` القديم إلى worktree منفصل ثم يشغّل `verify.py` عليه مباشرة. لذلك ظل الملف التاريخي القديم مثل `2026-07-17.json` موجوداً داخل worktree المستورد وفشل البناء قبل أن يعمل Workflow التحديث ويصلح الفرع المنشور.

كما أن الفرع المنشور القديم لا يحتوي بعد على `data/update-manifest.json` وتوقيعه، ولذلك كان سيظهر خطأ ثانٍ مباشرة بعد إزالة alias القديم.

## الإصلاح

- يشغّل `build.yml` أداة `clean_legacy_calendar_snapshots.py` داخل worktree المنفصل قبل نسخ البيانات، في كل من Debug وRelease.
- يستخدم Debug الخيار `--allow-missing-manifest` فقط لتوافق الانتقال من فرع قديم لا يحتوي **لا الملف ولا التوقيع**.
- إذا وُجد ملف Manifest بلا توقيع، أو توقيع بلا ملف، أو Manifest غير صالح، يفشل التحقق كالمعتاد.
- لا يستخدم Workflow Update ولا فحص النشر الرسمي هذا الاستثناء؛ يبقيان صارمين.
- يعيد Update تنظيف شجرة النشر المجمعة قبل التوقيع والتحقق كطبقة حماية ثانية.

## التطبيق

استخرج حزمة R18.2 في جذر المستودع، ارفع الملفات إلى `main`، ثم شغّل Workflow **Update** بوضع `update`. بعد نجاحه سيصبح فرع `verified-data` نظيفاً ويحتوي Manifest موقّعاً، ولن يعود Debug بحاجة فعلية لمسار التوافق القديم.
